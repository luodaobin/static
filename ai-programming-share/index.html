它到底在做什么

大模型的本质：概率猜字机器

01大模型根据前文，一个 token 一个 token 地预测后续内容。
02它本质上是在“续写”，只是规模足够大，表现出了知识、推理和代码能力。
03所以它很强，也会犯“看起来很合理但其实错误”的问题。
前文输入需求、代码、历史对话
→
概率分布下一个 token 的可能性
→
选择 token继续生成
关键点：它不是从数据库里“查答案”，而是在上下文约束下生成最可能的续写。
强能力与幻觉来自同一个机制
Token by Token
Context · Stateless
04 / 18
它不会天然记住你

大模型是无状态的

01模型服务器不会天然记住你是谁、刚才说过什么。
02每次交互，都要把历史消息重新发给模型。
03模型输出后，又会追加到下一次输入中。
04所谓“记忆”，通常是客户端不断维护 messages。
记忆在客户端，不在模型里

05 / 18
上下文不是免费的

Context 与 Context Window

01一直累积发送给模型的 messages，就是上下文 context。
02context window 是模型一次能读取的最大上下文长度。
03对话越长，token 消耗越快；满了以后只能压缩。
04多次压缩后，容易信息丢失、注意力丢失、幻觉变多。
messages = [
  {"role": "system", "content": self.render_system_prompt(react_system_prompt_template)},
  {"role": "user", "content": f"<question>{user_input}</question>"}
]

while True:
  content = self.call_model(messages)
  # ...处理 content
  # 等待用户输入或调用工具产生新的输入
  # ...产生新的请求数据 new_msg
  messages.append({"role": "user", "content": new_msg})
上下文满了以后，只能做压缩和取舍

为什么对话可以搬家

为什么可以在模型厂商之间切换

01因为大模型无状态。
02只要客户端保存完整上下文，就可以把同一份 messages 发给另一个模型。
03所以 Claude Code 对话到一半额度不够时，可以通过 CCSwitch 从 Anthropic 切到 DeepSeek。
04模型不同，能力、风格、工具适配会有差异，但机制上可迁移。
迁移的是上下文，不是模型记忆

09 / 18
不是天然会，而是被约束

Agent 的提示词约束

01Agent 并不是模型天然就会按流程工作。
02实际是通过很长的 system prompt 约束模型行为。
03要求输出特定结构，例如 thought、action、observation、final_answer。
04输出 action 后停止，等待真实工具返回 observation。
<system>
你是一个编程 Agent。
必须按下面协议输出：

<thought>
  简要说明下一步判断
</thought>

<action name="read_file">
  {"path": "src/app.ts"}
</action>

输出 action 后立刻停止，
等待工具返回 <observation>。

最终只在确认完成后输出：
<final_answer>...</final_answer>
</system>
格式约束 + 工具调用等待

Token Economy · Prefix Cost
10 / 18
新对话的第一口很大

Agent 的初始 Prompt 很贵

01主流 Agent 每次新对话都会带一段很长的前置约束文本。
02例如 Cline 的前置 prompt 约 45KB，转换后大约 12K 到 13K token。
03所以新开对话第一次交互就可能消耗 10K 左右 token。
04但这部分通常会命中缓存，实际费用和延迟会被优化。
缓存能优化成本，但上下文仍然在请求里

11 / 18
会话也需要边界

不要在一个会话里做所有事情

错误路径

长对话无限拖长，任务互相污染，上下文频繁压缩，最后模型开始忘掉关键约束。

推荐路径

阶段性总结 → 写入文件 → 新开会话 → 读取总结继续。让上下文保持干净。

长任务不是不能做，而是要像工程交接一样做。
上下文污染会慢慢吃掉质量
Summarize · Restart · Continue

12 / 18
先让它认识这片代码

让 AI 先理解现有系统

01AI 对已有系统一开始一无所知。
02每次都让它重新查代码，会浪费大量上下文和时间。
03推荐先让 AI 阅读项目，形成系统规格文档。
04后续做功能时，先让它读规格文档，再提出需求。
规格文档是高性价比上下文

写清楚结构，不必写得像代码

如何向 AI 描述需求

01静态类图可以转换成自然语言描述的 JSON 数据结构。
02交互接口要说清输入、输出、状态变化和边界条件。
03AI 往往能生成合理的类名、字段名、协议名称。
04重点不是“写得像代码”，而是结构清楚、约束明确。

模糊

做一个审批流，能驳回，也能通知人，权限别出问题。

结构化

{
  "entity": "ApprovalRequest",
  "states": ["draft","pending","approved","rejected"],
  "actions": {
    "submit": "draft -> pending",
    "reject": "pending -> rejected"
  },
  "guards": ["owner", "approver"]
}
自然语言也可以很精确

可以许愿，但上线前必须清醒

许愿式编程的边界

适合

工具类代码、一次性脚本、原型验证、低风险自动化。

谨慎

核心业务架构、高风险线上逻辑、安全、权限、资金相关代码。

低风险 / 一次性

可以大胆让 AI 做，人工抽查结果。

高风险 / 一次性

可以辅助，但要明确边界和回滚。

低风险 / 长期

关注可维护性，别留下看不懂的实现。

高风险 / 长期

必须设计、评审、测试、灰度，一个都不能省。

一次性 → 长期维护
低风险 → 高风险
