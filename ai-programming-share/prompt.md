第 2 页：我对 AI 编程的核心理解
内容：
- AI 不是魔法，而是一个“极强的概率猜字机器”
- 它无状态，但拥有庞大的知识储备和推理能力
- Agent 的本质，是把模型接入工具和环境形成循环
- 真正高效的使用方式，是管理上下文、明确任务边界、持续 review
视觉建议：用 4 个关键词卡片：概率、上下文、工具、Review。

第 3 页：大模型的本质：概率猜字机器
内容：
- 大模型根据前文，一个 token 一个 token 地预测后续内容
- 它本质上是在“续写”，但因为训练数据和模型规模足够大，表现出知识、推理、代码能力
- 所以它既强大，也会犯“看起来很合理但其实错误”的问题
图示建议：前文输入 → 概率分布 → 选择下一个 token → 继续生成。

第 4 页：大模型是无状态的
内容：
- 模型服务器不会天然记住你是谁、刚才说过什么
- 每次交互，都要把历史消息重新发给模型
- 模型输出后，又会追加到下一次输入中
- 所谓“记忆”，本质上通常是客户端不断维护 messages
视觉建议：画出 client messages 数组不断增长并发送到 model server 的流程图。

第 5 页：上下文 Context 与 Context Window
内容：
- 一直累积发送给模型的 messages，就是上下文 context
- context window 就是模型一次能读取的最大上下文长度
- 对话越长，token 消耗越快
- 上下文满了以后只能压缩
- 多次压缩后容易出现信息丢失、注意力丢失、幻觉
需要展示以下伪代码片段：

messages = [
  {"role": "system", "content": self.render_system_prompt(react_system_prompt_template)},
  {"role": "user", "content": f"<question>{user_input}</question>"}
]

while True:
  content = self.call_model(messages)
  # ...处理 content
  # 等待用户输入或调用工具产生新的输入
  # ...产生新的请求数据 new_msg
  messages.append({"role": "user", "content": new_msg})

图示建议：消息列表越来越长，最后触发“压缩”。

第 6 页：为什么可以在模型厂商之间切换
内容：
- 因为大模型无状态
- 只要客户端保存了完整上下文，就可以把同一份 messages 发给另一个模型
- 所以 Claude Code 对话到一半额度不够时，可以通过 CCSwitch 从 Anthropic 切到 DeepSeek
- 模型不同，能力、风格、工具适配会有差异，但机制上是可迁移的
视觉建议：同一份 Context 分别发往 Claude / DeepSeek / OpenAI 的示意图。

第 7 页：什么是 Agent
内容：
- 最初的大模型主要是聊天：我问你答
- Agent 是在模型外面包了一层工作循环
- 它可以思考、调用工具、观察结果，再继续下一步
- 编程 Agent 的核心不是“更会聊天”，而是“能和真实环境交互”
关键词：Thought、Action、Observation、Final Answer。

第 8 页：ReAct 范式
内容：
- ReAct = Reasoning + Acting
- 核心循环：Thought → Action → Observation
- 它把推理过程和外部工具调用统一起来
- Code Interpreter、Devin、Claude Code、Codex、Cline 等编程 Agent 都能看到类似思想
视觉建议：用循环图展示 Thought → Action → Observation → Thought。

第 9 页：Agent 的提示词约束
内容：
- Agent 并不是模型天然就会按流程工作
- 实际上是通过一段很长的 system prompt 约束模型行为
- 例如要求它必须输出 <thought>、<action>、<observation>、<final_answer>
- 输出 action 后停止，等待真实工具返回 observation
展示一个简化版 XML 协议提示词片段，不要全部塞满，重点高亮“格式约束”和“工具调用等待”。

第 10 页：Agent 的初始 Prompt 很贵
内容：
- 主流 Agent 每次新对话都会带一段很长的前置约束文本
- 例如 Cline 的前置 prompt 约 45KB，转换后大约 12K 到 13K token
- 所以新开对话第一次交互就可能消耗 10K 左右 token
- 但这部分通常会命中缓存，实际费用和延迟会被优化
视觉建议：Prompt 前缀 + 用户问题 + 工具说明，一起组成第一次请求。

第 11 页：不要在一个会话里做所有事情
内容：
- 一次会话里塞太多任务，会污染上下文
- 对话次数过多会频繁触发上下文压缩
- 多次压缩之后，模型可能忘掉关键约束
- 更好的方式：阶段性总结 → 写入文件 → 新开会话 → 读取总结继续
视觉建议：错误路径：长对话无限拖长；推荐路径：阶段总结文件接力。

第 12 页：让 AI 先理解现有系统
内容：
- AI 对已有系统一开始一无所知
- 每次都让它重新查代码，会浪费大量上下文和时间
- 推荐先让 AI 阅读项目，形成系统规格文档
- 后续做功能时，先让它阅读规格文档，再提出需求
视觉建议：代码仓库 → 系统规格文档 → 功能开发会话。

第 14 页：如何向 AI 描述需求
内容：
- 静态类图可以转换成自然语言描述的 JSON 数据结构
- 交互接口可以用自然语言描述清楚输入、输出、状态变化和边界条件
- AI 往往能生成合理的类名、数据库字段名、协议名称
- 重点不是“写得像代码”，而是“结构清楚、约束明确”
视觉建议：左边是模糊需求，右边是结构化 JSON / API 描述。

第 15 页：AI Agent 的定位
内容：
- 把 AI Agent 当成一位经验丰富的代码工程师
- 它能处理繁琐编码细节、边界条件、复杂算法
- 但它有时会为了达成目的不择手段
- 可能不够关注效率、架构层次和长期可维护性
- 所以 review 是非常必要的
视觉建议：AI 写代码，人类做架构判断和 review 的协作图。

第 16 页：许愿式编程 Vibe Coding 的边界
内容：
- 适合：
  - 工具类代码
  - 一次性脚本
  - 原型验证

  - 核心业务架构
  - 高风险线上逻辑
  - 安全、权限、资金相关代码
总结句：可以许愿，但上线前必须清醒。
视觉建议：风险坐标图：一次性/长期维护，低风险/高风险。

第 17 页：实践建议总结
内容：
1. 先理解机制，再决定怎么用
2. 控制会话边界，避免上下文污染
3. 长任务阶段性总结，必要时新开会话
4. 让 AI 先读规格文档，再做需求
5. 复杂功能用工作流，不要只靠一句话
6. AI 负责执行，人负责判断
视觉建议：6 条简洁 checklist。
